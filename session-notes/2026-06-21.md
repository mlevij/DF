## Iframe sizing vs. Leaflet zoom — two separate knobs

User reported the embedded maps (drought monitor, Sentinel) were too tall for a smaller monitor. Scaled down the CSS:
- `.map-embed` 600px → 480px
- `.map-embed-tall` 1100px → 720px

This alone didn't fix the "boundaries cut off" look — **shrinking the container doesn't change how much geographic area is visible at a fixed Leaflet zoom level**. At a constant zoom, fewer vertical pixels just shows less ground area, re-introducing the original cutoff problem from the other direction. Fix: reduced the actual `setView()` zoom argument from `8.5` → `7.5` in both `drought-monitor.html` and `sentinel-ratio-monitor.html` to compensate — zooming out a notch so the full intended area fits in the smaller container.

**The exact line changed each time**: `}).setView([38.1, -108.0], 7.5);` — second argument is zoom level.

## Caching confusion ("the zoom always comes out the same")

User repeatedly saw no visible change after pushes. Diagnosed: this is most likely **iframe sub-resource caching** specifically — `index.html` embeds `drought-monitor.html`/`sentinel-ratio-monitor.html` via `<iframe src="...">`, and a hard-refresh of the parent page doesn't reliably force a fresh fetch of the nested iframe document in all browsers. Recommended testing by opening the iframed page directly in its own tab; offered a cache-busting query string (`?v=2` etc. on the iframe `src`) as a future mitigation if it keeps recurring, not yet implemented.

### Surface / root-zone / groundwater — raw GEE layers added to the .js script
User asked whether to add surface/root-zone/groundwater soil moisture breakdowns alongside the GRACE TWS chart. Resolved:
- The already-downloaded GRACE-DA NetCDFs (`GRACE_NC4/`, 1230 cached weekly files) contain ONLY percentile variables (verified via `nc_open` → `gws_inst`, `rtzsm_inst`, `sfsm_inst` — no raw value band), so "recompute raw from the percentile files" isn't possible — a percentile can't be inverted back to a physical value.
- Instead added raw GEE-derived layers to `gee/soil_moisture_acquisition.js`: broke the existing GLDAS NOAH compositing into `gldas_surface` (0–10cm) and `gldas_rootzone` (0–10/40/100cm sum) bands alongside the existing `gldas_tws` total, all as anomalies vs. the same 2004–2009 baseline as GRACE — so all three share one cm-LWE axis. Added three new `Export.table.toDrive` CSV tasks: `gldas_surface_swco_timeseries`, `gldas_rootzone_swco_timeseries`, `grace_gws_swco_timeseries` (the GLDAS-corrected groundwater variant — distinct from the live map's `gws_inst` percentile toggle).
- Naming asymmetry explained to user: `gldas_*` = pure GLDAS model output (no satellite gravity involved); `grace_gws` = GRACE satellite TWS measurement minus the GLDAS model correction term — GRACE is the primary source there, GLDAS is just the correction. Also clarified there is no purely-satellite-measured, depth-resolved soil moisture product — surface/root-zone is inherently model output (GLDAS), not a direct retrieval, unlike Sentinel-1's actual radar backscatter.
- Planned: once these 3 CSVs land, build a 3-line companion chart (surface/root-zone/groundwater anomaly over time) to sit next to the existing TWS chart, telling a "response speed by depth" story (surface fast/jagged, groundwater slow/smooth). Also need to add a line to the GRACE chart caption explaining the 2004–2009 baseline (NASA's fixed mascon-product reference period, chosen pre-2010s Colorado River Basin megadroughts — not a rolling/30-yr normal).

### Performance bug found and fixed in the .js script (mid-session)
User reported the 3 new CSV export tasks (surface/root-zone/GWS) were taking an extremely long time to run in GEE — much longer than the original 2 CSVs. Root-caused to two separate anti-patterns introduced while adding the GLDAS layers:
1. `graceGWS` did `monthlyGldas.filter(ee.Filter.eq('year_month', ym)).first()` **inside** a `.map()` over every GRACE image — forces EE to fully re-evaluate `monthlyGldas`'s entire month-by-month construction once per GRACE image (~96×96 recomputation). Fixed with a proper `ee.Join.saveFirst` (year_month match) between `graceAnomalies` and `monthlyGldas` instead.
2. `monthlyGldas` itself was built by filtering the ~23,000-image 3-hourly GLDAS-over-CONUS collection by a `year_month` **string-equality** property, once per each of ~96 months — can't use EE's time index, forces a near-full scan per month. Fixed by switching to `filterDate(monthStart, monthEnd)` per month (parsed from the `ym` string via `ee.Date.parse`), which EE can index directly. This affects all three new CSV exports, not just GWS.
