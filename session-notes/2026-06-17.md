## Environment debugging: "kernel died" on import

### Root cause #1 ‚Äî confirmed with evidence
User runs the notebook via Anaconda Navigator ‚Üí environment **`arcpro_clone`** (`C:\Users\mlevij\AppData\Local\Programs\ArcGIS\Pro\bin\Python\envs\arcpro_clone\python.exe`), the ArcGIS Pro Python clone env.

Isolated each import in a separate process (`python -u -c "import X"`, checked exit codes):
- numpy, matplotlib (+Agg backend), pyplot, imageio, imageio.v3, requests ‚Äî all OK
- **`import rasterio` alone crashes with exit code `-1073741819` = `0xC0000005`** (Windows access violation)

Cause: `rasterio` (pip, v1.3.10) bundles its own GDAL-linked native DLLs, which conflict with ArcGIS Pro's own bundled GDAL already loaded in that environment/process. This matches the warning already present in the script's own comment ("prevents 0xC0000005 on Windows" ‚Äî that comment was about a *different*, already-fixed matplotlib-backend issue from a prior session; the rasterio crash is separate and was not previously diagnosed).

### Fix ‚Äî switch environments
Found `gee_drought` conda env (`C:\Users\mlevij\AppData\Local\anaconda3\envs\gee_drought`) ‚Äî set up earlier in the semester for this kind of work, not yet used for this script. Had jupyter/matplotlib/numpy/imageio/requests but no rasterio/gdal.

Installed via `conda install -n gee_drought -c conda-forge rasterio -y`. Confirmed all notebook imports work cleanly afterward in this env (only a harmless `GDAL_DATA`/`gdalvrt.xsd` warning, no crash).

### Root cause #2 ‚Äî new issue introduced by the rasterio install
After switching to `gee_drought`, launching `jupyter notebook` itself crashed:
```
ssl.SSLError: [ASN1: NOT_ENOUGH_DATA] not enough data (_ssl.c:4057)
```
raised from `tornado.netutil` building a default SSL context at import time (`ssl.create_default_context()` ‚Üí `load_default_certs()` ‚Üí enumerating the Windows certificate store).

**Confirmed root cause with evidence** (not just inferred):
- `conda list -n gee_drought --revisions` showed: env created 2026-03-16 with `openssl-3.5.5` (defaults channel); the rasterio install today bumped it to `openssl-3.6.3` (conda-forge).
- Reproduced minimal failure: bare `ssl.create_default_context()` (no tornado) crashes in `gee_drought` (OpenSSL 3.6.3) but succeeds in `arcpro_clone` (OpenSSL 3.0.18) with the identical call.
- Conclusion: OpenSSL 3.6.3's stricter ASN.1 parser chokes on a cert already present in the Windows certificate store (likely a CSU network/VPN/AV root cert) that older OpenSSL versions tolerate. User confirmed they'd never hit this before ‚Äî environment had never had rasterio/this OpenSSL version before today.

**Fix applied**: `conda install -n gee_drought "openssl=3.5.5" -y` (downgrade back to the pre-rasterio-install version). Verified both `ssl.create_default_context()` succeeds AND `rasterio` still imports cleanly. User confirmed Jupyter launched successfully after this fix.
