## Abandoned rasterio entirely — switched GRACE pipeline to netCDF4

Rather than keep fighting the rasterio/GDAL/OpenSSL conflict from 2026-06-17, switched the whole approach: GRACE-DA is distributed as `.nc4` files (one file per week contains all three percentile variables — `sfsm_inst`, `rtzsm_inst`, `gws_inst`), so `netCDF4` reads it directly with no GDAL involved at all. Confirmed `netCDF4` imports cleanly in the `gee` env; sidesteps the whole DLL saga.

**Important data-nature correction**: GRACE-DA is NASA GSFC's *Data Assimilation* drought indicator — a land-surface model (Catchment) continuously run and nudged toward consistency with GRACE/GRACE-FO observations at each overpass, **not raw satellite retrieval**. That's also why it covers 2002–present continuously despite the real ~2017–2018 GRACE→GRACE-FO mission gap (the model just keeps running through the gap). The GEE script's separate GRACE-FO Mascon section (raw satellite TWS anomaly, no model) is genuinely raw, but only continuous from 2018+ — hence that script's `startYear = 2018`. User wants the raw Mascon data eventually; for now sticking with GRACE-DA.

## Colorado-region rendering — long centering/aspect debugging saga

`render_grace_frames_colorado.py` went through several rounds of "still off-center / still square" before landing on a working approach. Root causes, in order discovered:

1. **UTM distortion**: converting a lon/lat bounding box via `ax.set_extent()` distorts badly in UTM (worse the further from the central meridian) — pushed the state off-center. Fixed by computing extent directly from the *projected* polygon's own bounds instead.
2. **`fig.colorbar(im, ax=ax, ...)` permanently corrupts the axes' stored "original position"** that matplotlib's aspect/datalim logic relies on internally — no later `set_aspect()` call could undo this, regardless of order. Confirmed by direct pixel measurement (objectively measuring left/right margins in the rendered PNG, not just eyeballing) showing the exact same broken result whether `set_aspect` was called before or after colorbar. **Fix**: give the colorbar its own dedicated axes (`fig.add_axes()` + `cax=`) instead of letting `colorbar()` auto-shrink the main one.
3. **`ax.imshow()` itself calls `set_aspect()` internally**, silently overriding anything set earlier — `adjustable='datalim'` needs to be set immediately after axes creation, before `imshow`/`add_feature`/anything else touches the axes.
4. **Hand-picked lon/lat center point still landed off-center** (93px asymmetry, objectively measured) — UTM's meridian convergence distorts a guessed lon/lat center away from the polygon's true projected center. Fixed by using the actual polygon's own bounds midpoint in projected meters instead (28px asymmetry, much closer).

General lesson reinforced: when "I fixed it" claims don't match what the user sees, get objective pixel measurements (PIL + numpy, scan rows for non-white content bounds) rather than trusting visual inspection on either side — this is what actually found bug #2 above, which visual inspection alone had missed twice.
